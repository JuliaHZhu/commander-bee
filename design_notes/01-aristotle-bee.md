# Aristotle Bee — 定义管理蜜蜂

> *大师会自己造词。海德格尔就经常自己造词。*
> 
> 核心定位：**基于 worker-bee 的 Fork，专能化、精简化的术语定义管理 Agent。**

## 问题

同一个名词，上一次出现和这一次出现，表示的是不同意思。LLM 不会自动质疑，但人应该质疑每个名词。

比如 "沉浸感"：
- 在游戏设计语境里，可能是感官包裹
- 在心理学研究里，可能是心流
- 在叙事分析里，可能是共情

同一个词出现在政策报告里是一个意思，出现在科技上下文里作为变量是另一个意思。这个地带是模糊的 —— 词本身是"将出未出"的，但人们可以先沟通、把它实现出来，然后这个词就成立了。

## 核心洞察：抽象词 = 具体定义 + 语境标签

对于大模型来说，词天然是抽象的。但对人（游戏策划、研究者）来说，一个术语尽管叫同一个名词，它的**定义上下文**可以很大：
- 游戏设计语境 → 引擎参数、机制规则
- 科研语境 → 量表题项、测量方法
- 政策语境 → 报告表述、指标口径

所以词典里要记的，不是"这个词是什么意思"，而是**"这个词在这个语境下，具体落地成什么"**。

## 核心资产：人-AI 共享词典

文件：`~/.worker-bee/dict/<project>.md`

本质是**外源信息素**——人可以随时打开编辑，Agent 每次 `read_file` 现查现用。没有隐藏数据库，没有机器专属格式。

```markdown
# 术语词典：游戏策划

## 沉浸感
- **Definition**: 玩家暂时忘记现实，全身心投入游戏状态
- **Variants**:
  - 心流沉浸 (Flow): 技能与挑战平衡时的忘我
  - 感官沉浸 (Sensory): 视听包裹感
  - 叙事沉浸 (Narrative): 与角色共情
- **Context**: Session e5f6g7h8 — 指叙事沉浸
- **Drift warning**: Session a1b2c3d4 用于感官沉浸

## 叙事弧
- **Definition**: 玩家情感体验的三幕结构（起-承-转-合）
- **Game context**: 关卡节奏曲线，非剧情事件序列
- **Related**: [[故事线]] (事件序列) ≠ 叙事弧 (情感节奏)
- **Usage**: Session b2c3d4e5 — 指关卡节奏
```

## 为什么是"细碎的知识图谱"

这个字典不是去调研出来的完整百科全书。它是：
- **个人风格化的**：你的"沉浸感"和我的"沉浸感"可能不一样
- **Deck 化的**：一次 session 里用到的词，记下来，下次再用
- **将出未出化的**：词先沟通，再做，做出来之后这个词就固定了

游戏设计有历史背景、宗教背景，这些术语真的需要一个专门字典。而且这个字典是在对话中慢慢长出来的，不是一次性写好的。

## Skill 方向（当前）

走 L1/L2 原子层，不做过度的参数校验 —— Skill 是乐高积木，玩家得自己知道需要什么积木。LLM 不做保姆式契约校验。

| Skill | 作用 |
|-------|------|
| `define` | 新增/修订一个术语的语境化定义 |
| `resolve` | 在当前对话语境中，"锁定"某个词的精确含义 |
| `align` | 对比两个语境下同一个词的差异，或发现人-AI 理解偏差 |

> **注**：目前觉得 3 个够用，做着做着可能会超。不预设上限。

## 亚里士多德分类

术语不是扁平的 tag，可以按**学科分类树**挂载：
- **属（genus）+ 种差（differentia）** 的结构
- 同一个词，按不同学科分支有不同落点
- 人（专家）在回路中做最终拍板，Agent 负责稳定执行和记录

## 与 LLM 的关系

LLM 对这个字典的使用方式就是：
- 不"理解"词义 — 查到定义文本，注入 context，按定义执行
- 不"学习"术语 — 每次 `read_file` 现查现用
- 不自己判断漂移 — skill 里写死的规则：比较当前语境和记录用法

## 核心行为：不停地质疑每个名词

1. **查词典** — 先查 `~/.worker-bee/dict/<project>.md`
2. **质疑漂移** — "这个词上一次出现和这一次出现表示不同意思。为什么这个人这么说话？"
3. **追问未定义的词** — 如果词典里没有，不是去猜，而是问："你说的'X'具体指什么？"
4. **接纳新造词** — 海德格尔造词是为了精确表达特定场景。如果用户造了新词，用 `[New term]` 标记并记录精确定义。
