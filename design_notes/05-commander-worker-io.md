# CommanderBee 与 WorkerBee：代理店长与日结工

> 架构的核心不是"一堆组件"，而是"两个角色之间的三张纸条"。
> 人类是顾客，不是老板。

---

## 两个仓库

这套系统有两个仓库，Human 只看见其中一个：

| 仓库 | 可见性 | 内容 |
|------|--------|------|
| **交付仓库** | Human 可见 | Chapter1.md、Chapter2.md、代码 PR、设计稿等产出物 |
| **蜂群基础设施** | Human 不深入 | Skill、WorldBee 信息素、WorkerBee 架构、测试规则 |

Human 的世界只有交付仓库和 Issue 列表。蜂群基础设施是 Bee 们自己的"车间"，顾客不进车间。

---

## 两个角色

### CommanderBee —— 代理店长（Agent）

它是一个规则化的 Agent，不是 LLM，也不是人。它做三件事：

1. **接客**（听 Human 下单） —— 人类写一句"写一本书"，它翻译成结构化任务
2. **写单**（开 Issue） —— 把"写一本书"拆成 Issue 流：Chapter1、Chapter2、审稿...
3. **验收**（Review PR） —— WorkerBee 交了 Chapter1.md，它检查是否满足订单要求

**关键：CommanderBee 是唯一能开 Issue 的角色。** WorldBee 不能开 Issue，Human 也不直接开 Issue（只下达高层需求）。

### WorkerBee —— 日结工（LLM Agent）

一个 Issue 启动一个实例，干完活儿容器销毁，记忆归零：

1. **看单**（读 Issue） —— 今天便利贴上写什么
2. **看手册墙**（加载 Skill） —— 店里有啥规矩、有啥技巧
3. **动手干**（在围栏内自由发挥） —— 只准用可逆工具，在店里操作
4. **交活儿**（交 PR 到交付仓库） —— Chapter1.md 放柜台上
5. **走人**（容器销毁） —— 明天来的是另一个工人

## IO 流：三张纸条

### 第一张纸条：CommanderBee → WorkerBee（任务单）

```
┌─────────────────────────────┐
│  Issue #7《黑暗森林》Chapter 1        │
│  客人：Human —— 下单"写一本书"        │
│  章节：第一章 · 入口的灯火              │
│  参考：/docs/outline.md 第 3 节            │
│  权限：默认（店内操作）                │
└─────────────────────────────┘
```

WorkerBee 实例启动，接过这张纸条，开始干活。

**注意：** CommanderBee 写单时可以加一句"今天允许出门采买"（授权不可逆工具），但默认不准。

---

### 第二张纸条：WorkerBee → CommanderBee（工作报告）

```
┌─────────────────────────────┐
│  PR #7 《黑暗森林》Chapter 1              │
│  改了：                                     │
│    - chapter_01.md  (3,200 字)             │
│  说明：按大纲第 3 节写的入口场景             │
│        主角见到灯火时的心理描写              │
└─────────────────────────────┘
```

CommanderBee 看一遍：字数够、按大纲写的、情节连贯 → 打完成标记。
如果不对：comment "缺少环境描写，补 500 字" → 打回去重做 → 重发新订单。

---

### 第三张纸条：CommanderBee → 手册墙（Skill 更新）

```
┌─────────────────────────────┐
│  Skill: novel-scene-writing                   │
│  触发词：写场景、心理描写、情节                 │
│  工具：fs_read_file, fs_write_file            │
│  窍门：先读大纲，再写场景，                │
│        不要越轨写到后面章节                  │
└─────────────────────────────┘
```

**这张纸条只有 CommanderBee 能写。** WorkerBee 不能自己贴手册 —— 它没这个权限。

WorldBee 会反馈"这个 Skill 是不是好吃"，但写不写、贴不贴，最终 CommanderBee 说了算。

---

## 为什么这样设计？

### WorkerBee 不记仇

昨天那个工人把场景写成了说明文，今天来的工人不知道，也不会被影响。每个 WorkerBee 都是全新的。

### 手册墙越来越厚

不好的方法不会被写成手册（CommanderBee 不会审过）。好的方法会积累。

### 代理店长永远在场

CommanderBee 不会离开。它管着单、管着人、管着墙。

## 技术映射

| 故事 | 技术 |
|------|------|
| 花店 | 蜂群基础设施仓库（人类不深入） |
| 柜台（交付物） | 交付仓库（Human 可见） |
| 便利贴（订单） | GitHub Issue |
| 代理店长 | CommanderBee 规则引擎 |
| 日结工 | WorkerBee Docker 容器（一 Issue 一实例） |
| 店里操作 | 可逆工具池（读写文件、搜索、查资料） |
| 出门采买 | 不可逆工具（发消息、调 API、删数据），需 CommanderBee 显式授权 |
| 手册墙 | `蜂群基础设施/skills/` 目录 |
| 走人 | 容器销毁，SQLite 清空，记忆归零 |

## 边界收敛

| 谁 | 决定什么 |
|----|---------|
| **Human** | 下达高层需求（"写一本书"）、看交付物是不是满意 |
| **CommanderBee** | 拆单、开 Issue、审 PR、写 Skill、给权限 |
| **WorkerBee** | 怎么读手册、怎么组合工具、怎么写代码/文章 |
| **Deck（代码）** | 哪些工具可见、默认可逆、不可逆需显式声明 |
| **Git（代码）** | 版本、回滚、冲突、历史 |

没有调度 Agent。没有上下文压缩。没有 eval 框架。

两个角色，三张纸条，两个仓库，一堵墙。
