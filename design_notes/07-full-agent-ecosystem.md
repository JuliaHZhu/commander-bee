# 全 Agent 蜂群：分层架构与双仓库

> 不是人类领导蜂群。是四种不同智能体的协同 —— 有的靠规则，有的靠沉积，有的靠幻觉。
>
> 人类是顾客，只看柜台。

---

## 一、四个角色与两个仓库

```
┌──────────────────────────────────────────────────────────────────────────────┐
│  Human —— 顾客 / 总编                              │
│  行为：下单、看柜台、偶尔罚工                              │
└───────────────────────────────────────────────────────────────────────────────┘
                              │
                              ▼ 下单："写一本书"
┌───────────────────────────────────────────────────────────────────────────────┐
│  CommanderBee —— 代理店长（规则 Agent）                    │
│  数据来源：项目管理系统（Linear/Jira/飞书/飞书项目/Notion）    │
│  行为：结构进结构出，不幻觉                                │
│  职能：唯一能开 Issue 的角色                                  │
└───────────────────────────────────────────────────────────────────────────────┘
                              │
                              ▼ 生成 Issue
┌───────────────────────────────────────────────────────────────────────────────┐
│  WorkerBee 蜂群 —— 日结工（LLM Agent，一 Issue 一实例）  │
│  数据来源：外源信息素（Skill / Warning / Route）             │
│  行为：自由推理，但被围栏锁住（Deck + Git + 容器）         │
└───────────────────────────────────────────────────────────────────────────────┘
                              │
                              ▼ 产出 PR
                              │
        ┌────────────────────────┐
        │   交付仓库            │  ← Human 可见
        │   Chapter1.md         │
        │   Chapter2.md         │
        └────────────────────────┘
                              │
                              ▼ 测试数据 + 执行日志
┌───────────────────────────────────────────────────────────────────────────────┐
│  WorldBee —— 免疫系统（规则监督）                         │
│  数据来源：物理引擎 / 测试环境 / 性能监控 / 执行轨迹    │
│  行为：不审 PR，不下命令，只产生信息素                       │
│  职能：沉积环境，促使反思，但从不插手工单流                       │
└───────────────────────────────────────────────────────────────────────────────┘
                              │
                              ▼ 更新蜂群基础设施里的信息素
                              （skills/ / warnings/ / routes/）
```

### 两个仓库

| 仓库 | 见者 | 内容 |
|------|------|------|
| **交付仓库** | Human + CommanderBee + WorkerBee | Chapter1.md、代码 PR、设计稿等 |
| **蜂群基础设施** | Bee 们自己 + WorldBee | Skill、信息素、架构代码、测试规则 |

**Human 不进蜂群基础设施。** Human 的世界只有交付仓库和 Issue 列表。

## 二、分层的意义

为什么要分规则层、自由层、监督层？

因为它们的错误成本和运行方式不同：

| 层级 | 角色 | 错误成本 | 运行方式 |
|------|-------|----------|----------|
| 顶层 | Human | 最高 | 最少干预，偶尔下单 |
| 规则层 | CommanderBee / WorldBee | 中（系统级） | 规则引擎，结构进结构出 |
| 自由层 | WorkerBee | 最低（单体） | LLM 推理，自由组合 |

**CommanderBee 如果分错了单，影响是系统级的。** 所以它不能幻觉，必须靠规则。

**WorkerBee 如果干炸了，影响只是一个 PR 没通过。** 它的容器会被销毁，记忆不会留下。

**WorldBee 如果撒错了味儿，影响是环境级的。** 所有 Bee 都可能被带偏。但它的规则是物理/测试数据驱动的，不是 LLM 幻觉。

## 三、CommanderBee 是怎么工作的？

不是 LLM 写任务。是规则编排任务。

```
输入：项目管理系统的数据
│
│  例如：
│  - 需求标题："《黑暗森林》第一章"
│  - 优先级：P1
│  - 标签：小说, chapter
│  - 截止日期：2026-06-15
│
└───────────────────────────────────────────────────────────────────────────────┘
                              │
                              ▼ 规则引擎（不是 LLM）
┌───────────────────────────────────────────────────────────────────────────────┐
│  如果 标签包含 "小说" → 加载 creative-writing Skill 到 Deck │
│  如果 优先级 = P0 → 加载紧急协议工具（需人审）         │
│  如果 截止日期 < 3 天 → 打标签 "急件"                       │
│  ...                                                     │
└───────────────────────────────────────────────────────────────────────────────┘
                              │
                              ▼ 输出结构化 Issue
```

**CommanderBee 是一个规则引擎，不是 LLM。** 它的价值在于：
- 把人类写的需求（模糊）翻译成 Bee 能执行的指令（结构化）
- 不会因为 LLM 幻觉而把"修个错别字"误翻成"重写整本书"
- 是唯一能开 Issue 的角色，确保单口统一

## 四、WorldBee 是怎么工作的？

不是人类观察后写手册。不是审 PR。不是管理者。

WorldBee 是一个**独立运行的物理规则驱动的监督机制**，像免疫系统：

```
输入：测试环境 + 物理引擎 + 性能监控 + 执行轨迹
│
│  例如：
│  - WorkerBee #7 执行了任务 A，CI 报错：构建失败
│  - 物理引擎模拟：路径 X 的成功率 30%，路径 Y 的成功率 85%
│  - 性能监控：使用 Skill "web-scrape-v1" 的任务平均花费 120s
│
└───────────────────────────────────────────────────────────────────────────────┘
                              │
                              ▼ 规则引擎（不是 LLM）
┌───────────────────────────────────────────────────────────────────────────────┐
│  如果 CI 失败率 > 10% → 在 warnings/ 新增苦味标记          │
│  如果 路径 X 成功率 < 路径 Y → 更新 routes/ 优化路线        │
│  如果 Skill "web-scrape-v1" 耗时 > 60s → 提示更新 Skill 或降级  │
│  ...                                                     │
└───────────────────────────────────────────────────────────────────────────────┘
                              │
                              ▼ 输出结构化环境调整（信息素）
```

**WorldBee 也是一个规则引擎，不是 LLM。** 它的价值在于：
- 从客观数据中提取模式，不依赖人类观察
- 问题发生后自动反应（苦味标记），而不是等人发现
- 优化方向有数字作为依据（成功率、耗时、资源占用）
- 从不审 PR，从不下达命令，只是改变空气

---

## 五、WorkerBee 是唯一的 LLM 层

蜂群里只有 WorkerBee 用 LLM。

只有它们的错误是"低成本"的：
- 幻觉了 → 交的 PR 有问题 → 不 merge → 重发任务
- 毫无损失（除了 Token 成本）
- 不影响系统级状态
- 容器销毁，记忆归零

它们的自由被三层围栏锁住：
1. **Deck** —— 只能用装填好的工具
2. **Git** —— 所有产出进入版本控制
3. **容器** —— 执行完毕即销毁

而且它们从不知道彼此的存在，只闻得见味儿。

## 六、闭环：环境的自然选择

```
                ┌────────────────────────────┐
                │   CommanderBee 规则分发   │
                └────────────────────────────┘
                         │
                         ▼ Issue
                ┌────────────────────────────┐
    ┌───────┐  ←  │   WorkerBee 执行        │
    │ 信息素  │       └────────────────────────────┘
    │ 调整   │                │
    └───────┘                ▼ PR / 测试
         ↑                       │
         │                ┌────────────────────────────┐
         │                │   WorldBee 规则调优   │
         └──────────────└──────────────────────────────┘
```

这是一个自然选择的闭环：

1. CommanderBee 发单（根据项目系统规则）
2. WorkerBee 执行（根据环境信息素自由推理）
3. 测试/物理引擎收集结果
4. WorldBee 根据数据调环境（甜味、苦味、路线）
5. 下一轮 WorkerBee 在更好的环境里执行

**不需要 LLM “学习”。只需要环境 “变化”。**

好的路线自然被保留，坏的路线自然被标记。Bee 还是那些 Bee，但它们闻到的味儿变了。

## 七、人类在哪里？

Human 不是日常运营者，是"顾客"和"纠偏者"：

| 场景 | Human 做什么 |
|------|--------------|
| 下单 | 说"写一本书"，看 Chapter1 满不满意 |
| 运行中 | 偶尔看一眼交付仓库，不深入蜂群基础设施 |
| 异常 | 当 WorldBee 撒了太多苦味，所有 Bee 都在等审批时，手动清空 warnings/ |
| 进化 | 写新的 Skill（因为发现了新领域，不是因为测试数据说要写） |

**Human 不审每个 PR。Human 的界面是交付仓库。**

> **CommanderBee 是信使，负责发单。WorldBee 是农夫，负责改土壤。WorkerBee 是蜜蜂，负责采蜜。人类是顾客，偶尔看一眼柜台。**

> **蜂群越大，土壤越肥。土壤越肥，蜂群越强。这是对数增长的本质。**
