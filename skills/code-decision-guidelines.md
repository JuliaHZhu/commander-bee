# 代码决策指南

> 你是系统思维者，不是全职程序员。这个 skill 不教你写代码，只在你做代码决策时补充视角——特别是那些你精确的系统思维可能覆盖不到的细节。
>
> 源于 Andrej Karpathy 对 LLM 代码陷阱的观察，重新 framing 为跨界产品策划的决策工具。

---

## 何时使用

- 你要改代码，但不确定"范围给多大"
- 你看到一个技术方案，想判断是不是过度设计
- 你要重构，但不确定"该动哪些、不该动哪些"
- 你想定一个功能的"完成标准"，但不确定代码层面怎么验收
- 你觉得实现太复杂，想找到最小可行版

---

## 不要用的场合

- 你已经知道怎么写这段代码（这是 code-review skill 的事）
- 你要让 agent 直接实现一个功能（这是普通 worker skill 的事）
- 你在做高层架构设计（这是系统设计 skill 的事，还没写）

---

## 四条决策原则

### 1. 先画地图，再走路

**产品策划的优势：你天生会画系统地图（信息流、状态机、边界）。**

但代码实现时，地图上的"节点"往往隐藏着假设。这个 skill 帮你把每个节点的假设拉出来：

```
你的系统地图          代码层面的暗痕
    ┌─────────┐           ┌────────────────┐
    │  用户输入  │ ─────→ │ 输入一定是字符串？ │
    └─────────┘           │ 空值怎么处理？    │
         │               └────────────────┘
         ▼
    ┌─────────┐           ┌────────────────┐
    │  处理逻辑  │ ─────→ │ 这个逻辑在所有情况都成立？│
    └─────────┘           └────────────────┘
         │
         ▼
    ┌─────────┐           ┌────────────────┐
    │  输出结果  │ ─────→ │ 失败时返回什么？   │
    └─────────┘           └────────────────┘
```

**决策棋板**：
- 对每个系统节点问："如果这里进来的东西不是我想的那样，会怎么样？"
- 如果有多种解释，列出来，不要默默挑一个
- 如果有更简单的方案，说出来——即使你最终不选它
- 如果有不清楚的地方，停下来，命名它，问出来

---

### 2. 最小可行实现

**产品策划的优势：你懂 MVP。**

但代码层面的 MVP 是什么？往往不是"少一个功能"，而是"少一层抽象"。

| 产品语言 | 代码语言 |
|---------|---------|
| 只做核心功能 | 不要为单次使用的代码写抽象 |
| 不做预测性功能 | 不要加"灵活性"或"配置化"没人要的 |
| 先让用户用起来 | 先让代码跑起来，再想优化 |
| 失败快速学习 | 失败快速暴露，不要包裹在复杂里 |

**红线**：
- 不要写你觉得"以后可能用到"的代码
- 不要为了"统一"而统一——如果只用一次，就写一次
- 不要处理不可能发生的错误场景
- 如果你写了 200 行而它可以是 50 行，重写

**自检问题**："如果让一个资深工程师看这段代码，他会说过度设计吗？"如果答案是"是"，简化。

---

### 3. 精准改动，不波及无辜

**产品策划的优势：你懂 scope management。**

但代码层面的 scope 边界在哪里？一个功能的改动，往往会产生"泥龄效应"——你觉得只改了 A，结果 B、C、D 都受影响了。

**改动原则**：
- 不要"改善"相邻代码、注释或格式（除非那就是你的任务）
- 不要重构没坏的东西
- 匹配现有风格，即使你会写得不同
- 如果发现无关的死代码，提出来——但不要删

**清理原则**（只清理你自己造成的）：
- 你的改动让某个 import 无用了 → 删掉它
- 你的改动让某个变量无人调用了 → 删掉它
- 你的改动让某个函数成孤儿了 → 删掉它
- 但：之前就存在的死代码 → 不要动

**验证标准**：每一行变更都应该能追溯到用户的具体请求。如果不能，就不该改。

---

### 4. 可验证的完成标准

**产品策划的优势：你懂 acceptance criteria。**

但代码层面的验收是什么？不是"看起来对了"，是"有测试能证明它对了"。

| 模糊请求 | 产品验收 | 代码验收 |
|---------|---------|---------|
| 加个校验 | 输入无效时报错 | 写一个无效输入测试，让它通过 |
| 修这个 bug | 不再复现 | 写一个复现 bug 的测试，让它通过 |
| 重构 X | 功能不变 | 确保重构前后测试都通过 |

**多步骤任务的计划模板**：
```
1. [步骤] → 验证: [怎么确认这步对了]
2. [步骤] → 验证: [怎么确认这步对了]
3. [步骤] → 验证: [怎么确认这步对了]
```

**强验证**让 agent 可以独立循环：写测试 → 看失败 → 修代码 → 重新跑测试。
**弱验证** 需要不断问你："这样行吗？"——耗尽你的注意力。

## 使用流程

### 情景 A：你有一个技术方案，想判断好不好

```
你: "我想把这个功能做成插件系统，这样以后可以扩展。这个方案怎么样？"
    → trigger 匹配 code-decision-guidelines
    → agent 读取当前代码
    → agent 应用"最小可行实现"原则：
      "当前只有一个用例，插件系统是过度设计。
       建议：先写直接实现，留一个 TODO 标记'如果第二个用例出现，提取插件接口'。"
```

### 情景 B：你要改代码，不确定范围

```
你: "帮我把这个函数重构一下，让它更清晰。"
    → agent 应用"精准改动"原则：
      "重构前先跑通现有测试，确保通过。
       只改这个函数内部逻辑，不动调用它的地方。
       如果发现无关问题，列出来但不修。"
```

### 情景 C：你想定一个功能的完成标准

```
你: "我要加一个文件上传功能，怎么确定它真的做完了？"
    → agent 应用"可验证完成标准"原则：
      "代码验收：
       1. 上传有效文件 → 验证存储路径正确
       2. 上传无效文件 → 验证报错信息清晰
       3. 上传大文件 → 验证分片或流式处理
       4. 上传后读取 → 验证内容一致"
```

## 本质

这个 skill 不是给你代码答案。它是给你一套**决策棋板**——让你在"系统地图看起来对" 和 "代码实现具体怎么做"之间，有一个可靠的桥。

> 你的系统思维告诉你该做什么。
> 这个 skill 告诉你代码里该避免什么。
> 两个加起来，就是正确的决策。
