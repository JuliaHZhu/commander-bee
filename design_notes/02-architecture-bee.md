# Architecture Bee — 架构原型蜂

> *架构在脑袋里是将出未出到逐渐清晰化的过程。*  
> *目标：规约到不能规约。算法是效率和资源的量纲。*

## 定位

Architecture Bee 是 Worker Bee 的一个 **fork（专能化 Agent）**，不是单体 skill。它通过**多个 skill 的排列组合**，像小马过河一样逐步逼近未知架构问题的核心。

适用场景：
- 软件架构（模块分解、接口设计）
- 电影叙事结构（弧线、转折点）
- 书籍内容结构（章节骨架、认知递进）
- 任何需要将"将出未出"的模糊目标，逼出清晰骨架的场景

## 核心哲学

### 1. 规约到不能规约（Reduction to Irreducible）

架构一开始是宏观的、混沌的。要让它清晰化，必须一层一层往下剥——**rank / 降维 / 规约**，直到露出那个再也拆不动的核心约束。

### 2. 正交基底（Orthogonal Basis）

核心暴露后，结构必须是**高内聚、低耦合**的：
- 每个模块/组件 = 一个基底向量，只处理一个维度
- 无重叠 —— 两个组件共享责任 = 基底不正交
- 完整覆盖 —— 每个需求都有人负责
- 可组合 —— 通过接口/衔接点组合

### 3. Big O 作为资源量纲

Archetype（原型）暴露后，要考虑"怎么实现"。算法选择的标准就是**效率与资源**：
- 软件：时间复杂度、空间复杂度
- 电影：观众认知负荷、场景切换成本、时长预算
- 书籍：概念密度、阅读成本、认知递进梯度

## Skill 流水线（5 阶段）

Architecture Bee 由 5 个串联的 stage-skill + 1 个入口 skill 组成，通过**外源信息素 Markdown 文件**接力。

```
┌─────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│  1. capture-    │     │  2. decompose-   │     │  3. reduce-to-   │
│     intent      │────▶│      goal        │────▶│      core        │
│  捕捉原始冲动    │     │   目标展开分解    │     │  规约到不能规约   │
└─────────────────┘     └──────────────────┘     └──────────────────┘
                                                            │
┌─────────────────┐     ┌──────────────────┐               │
│ 5. evaluate-    │◀────│ 4. expose-       │◀──────────────┘
│   complexity    │     │    archetype     │
│  Big O 资源评估  │     │   暴露结构原型   │
└─────────────────┘     └──────────────────┘
```

### Stage 1: capture-intent — 捕捉原始冲动

- **触发**: 用户说"我想…"、"有个想法"、"如果…就好了"
- **输入**: 用户的原始表述（混沌状态，可能只有一句话）
- **产出**: `~/.worker-bee/arch/<project>/01-intent.md`
- **行为**: 不要拆解，不要问"怎么做"。只做一件事：把模糊的冲动锚定成文字，让"将出未出"的东西显形。

### Stage 2: decompose-goal — 目标展开

- **触发**: `01-intent.md` 存在
- **输入**: 读取 `01-intent.md`
- **产出**: `~/.worker-bee/arch/<project>/02-goals.md`
- **行为**: 把混沌拆成层级结构。每个子目标必须回答"为什么必须存在"。同时划定**反目标**（Anti-goals）——不做什么和做什么同等重要。

### Stage 3: reduce-to-core — 规约到不能规约（🔥 灵魂）

- **触发**: `02-goals.md` 存在
- **输入**: 读取 `02-goals.md`
- **产出**: `~/.worker-bee/arch/<project>/03-core.md`
- **行为**: 对每个子目标不断问 **"为什么"**，直到答案变成：
  - "因为这就是物理现实"
  - "因为这就是用户刚需"
  - "因为去掉它，整个东西就不存在了"
  
  用**移除测试**（If Removed Test）确认：去掉这个约束，目标还成立吗？No → 确认为核心。

### Stage 4: expose-archetype — 原型暴露

- **触发**: `03-core.md` 存在
- **输入**: 读取 `03-core.md`
- **产出**: `~/.worker-bee/arch/<project>/04-archetype.md`
- **行为**: 把核心约束翻译成结构原型：
  - 软件 → 模块分解 + 接口契约
  - 电影 → 叙事弧线 + 转折点衔接
  - 书籍 → 章节骨架 + 认知递进逻辑
  
  引入正交基底检验：删掉一个组件，其他是否还能自立？

### Stage 5: evaluate-complexity — 复杂度/资源评估

- **触发**: `04-archetype.md` 存在
- **输入**: 读取 `04-archetype.md`
- **产出**: `~/.worker-bee/arch/<project>/05-complexity.md`
- **行为**: 对每条实现路径做效率与资源的量能评估，给出取舍理由。

### 入口: architecture-bee — 编排器

- **触发**: `architecture`、`架构`、`结构`、`原型`、`设计`
- **行为**: 检查 `<project>/` 目录找到最新未完成步骤，调用对应 stage-skill 继续。用户可随时中断、修改 Markdown、再续上。

## 外源信息素接力格式

所有 skill 之间靠**文件命名约定**接力，纯文本、人可读、人可改：

```
~/.worker-bee/arch/
└── <project>/
    ├── 01-intent.md      ← capture-intent
    ├── 02-goals.md       ← decompose-goal
    ├── 03-core.md        ← reduce-to-core
    ├── 04-archetype.md   ← expose-archetype
    └── 05-complexity.md  ← evaluate-complexity
```

## 设计决策

| 决策 | 理由 |
|------|------|
| 拆成 5 个 skill，而非 1 个 | 小马过河：问题简单走到 Step 2 就够，问题深一路走到 Step 5。排列组合应对未知。 |
| 文件接力，非内存状态 | 外源信息素原则：人在回路，随时可改。 |
| 反目标（Anti-goals） | 防止范围蔓延，明确不做什么和做什么同等重要。 |
| 移除测试 | 核心约束的判定标准：去掉它，目标还成立吗？ |
| Big O 泛化 | 不仅软件，任何架构都有"资源与效率"的量纲。 |
