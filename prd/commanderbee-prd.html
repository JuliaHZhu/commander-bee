CommanderBee 产品需求文档

蜂群任务调度系统 · 基于NATS的分布式Agent协调器

DRAFT v1.0 2026-06-24

一、产品概述

1.1 产品定位

CommanderBee 是一个自动化任务调度系统，负责在多个Worker Bee实例之间派发任务、监控执行状态、验收成果质量。它不是一个独立的新框架，而是Worker Bee的特殊实例，通过不同的Skill集合和Cronjob配置实现监工职能。

核心价值

  • 解放人力：批量任务自动派发，无需手动分配
  • 质量保证：基于acceptance标准自动验收，不合格自动重派
  • 故障恢复：超时检测+心跳监控，Worker离线自动回收任务
  • 透明可控：所有状态存储在Markdown文件，人可读可编辑

1.2 目标用户

用户角色需求使用场景
研究者 批量处理论文、生成文献卡片 103篇论文 → 自动派发 → 验收lesson.html质量
开发者 多机器协同开发、测试、部署 Mac开发 + Linux服务器测试 + 云端部署
内容创作者 批量写作、翻译、审校 50篇博客草稿 → 自动润色 → 验收语法和风格

1.3 非目标

  • ❌ 不做实时聊天机器人（用Worker Bee）
  • ❌ 不做Web界面（Phase 3可选）
  • ❌ 不做跨语言Agent（只支持Python + Worker Bee架构）

二、功能需求

2.1 任务派发 (P0)

功能描述验收标准
Skill匹配 根据job的required_skills字段，从bees.json筛选有能力的Worker 匹配规则：Worker的skills列表必须包含所有required_skills
负载均衡 优先派发给current_jobs < max_concurrent的Worker 不会给满载的Worker派发新任务
活跃优先 优先派发给last_heartbeat最近的Worker 最近5分钟有心跳的优先
派发记录 写入job的owner字段和phase=dispatched job meta.md frontmatter更新
NATS通知 发送swarm.task.new-job消息到Worker payload包含{job_id, owner, meta_path}
边界条件：
  • 无匹配Worker时，job留在created状态，等待新Worker上线
  • 同一job不重复派发（除非超时回收）

2.2 心跳监控 (P0)

功能描述验收标准
心跳接收 Worker每30秒发送swarm.heartbeat.<bee-id> 更新bees.json的last_heartbeat字段
超时检测 心跳超过2分钟→标记status=silent Cronjob每60秒检测一次
任务回收 Worker离线且有current_jobs→清空job的owner job回到created状态，可重新派发
自动注册 首次心跳→自动在bees.json创建条目 包含bee_id、skills、max_concurrent

2.3 成果验收 (P0)

功能描述验收标准
读取标准 读job meta.md的acceptance字段（自然语言） 示例："三页互链完整 + 论证链清晰 + 无低级错误"
检查artifacts 读job/artifacts/目录的所有文件 检查文件数量、字数、关键内容
LLM判断 用LLM理解acceptance标准，判断是否合格 返回：Pass / Needs work / Incomplete
硬阈值 字数<一半 或 文件缺失→自动Incomplete 不问LLM，直接重派
验收报告 写入job的phase=reviewed + 报告内容 报告包含：通过/不通过、原因、建议

2.4 超时介入 (P1)

超时类型检测时间处理动作
accept超时 派发后10分钟未收到job-started 清空owner → 回到created
execute超时 执行中4小时未收到job-done 发送swarm.command.rescind → 回收任务
blocked介入 Worker发送swarm.event.blocked LLM判断：重派 / 等人工 / 修改job

2.5 状态汇报 (P2)

功能描述触发方式
即时查询 用户问"看看进度"→生成状态报告 手动触发
每日周报 Cronjob每天9:00自动发送 定时触发
报告内容 蜂群状态（在线/离线）+ 任务进度（待派发/执行中/已完成/阻塞）+ 最近完成Top5 + 需要注意的异常 LLM汇总生成

三、技术方案

3.1 架构复用

CommanderBee = Worker Bee + 专属Skill集合。共享五层工厂架构：

5F: 蜂群装备（NATS + Cron + SessionDB） 4F: Agent外壳（~/.commander-bee/配置） 3F: 协议内核（protocols.py + loop.py） 2F: 工具边界（Deck + Registry + Skills） 1F: 工具实现（swarm, file, terminal...）
组件Worker BeeCommanderBee
配置目录~/.worker-bee/~/.commander-bee/
Skillslark-messaging, code-review...commander-dispatch, commander-acceptance...
Job目录jobs/（接单）commander-jobs/（派单池）
NATS角色接收swarm.task.*发送swarm.task.*

3.2 NATS消息协议

✨ JetStream持久化：消息保留90天，listener重启不丢消息，支持durable pull consumer。

方向subjectpayload用途
CB→WB swarm.task.new-job {job_id, owner, meta_path} 派发新任务
CB→WB swarm.command.rescind {job_id, reason} 超时回收
WB→CB swarm.event.job-started {job_id, bee_id} Worker开始干活
WB→CB swarm.event.job-done {job_id, bee_id, summary} Worker完成任务
WB→CB swarm.event.blocked {job_id, bee_id, reason} Worker遇到阻塞
WB→CB swarm.heartbeat.<bee-id> {status, current_jobs, skills} 心跳+能力上报

3.4 Job Meta格式

---
job_id: JOB-047
title: 拆解论文 Baker2005
required_skills:
  - paper-lesson-generator
  - paper-contributor-network
acceptance: |
  - lesson.html存在且字数≥3000
  - contributors.html存在
  - 三页互链完整（lesson ↔ contributors ↔ full.md）
  - 论证链清晰（理论基础+构念+量表+发现）
owner: bee-mac
phase: executing
created_at: 2026-06-24T10:00:00Z
dispatched_at: 2026-06-24T10:02:00Z
started_at: 2026-06-24T10:03:00Z
---

## 任务描述

从 Baker2005-full.md 生成完整的 lesson.html 和 contributors.html。

~/Documents/fulltext/Baker2005-full.md

~/wiki/lessons/Baker2005/

四、实施计划

Phase 1: MVP (P0 - 1周)

  1. 复制Worker Bee配置到~/.commander-bee/
  2. 写3个Skill：commander-dispatch, commander-monitor, commander-acceptance
  3. 实现bees.json心跳注册逻辑
  4. 配置2个Cronjob：心跳监控(60s) + 派发扫描(120s)
  5. 手动测试：创建job → Commander派发 → Worker执行 → Commander验收

Phase 2: 自动化介入 (P1 - 3天)

  1. 实现commander-intervention Skill
  2. 超时回收逻辑（accept 10分钟 / execute 4小时）
  3. blocked消息处理

Phase 3: 人类友好界面 (P2 - 3天)

  1. 实现commander-report Skill
  2. 每日周报Cronjob（9:00发送）
  3. Web dashboard（可选，后期）

Phase 4: WorldBee接管 (P3 - 待定)

  1. WorldBee负责job文件的物理投放
  2. CommanderBee只发敲门铃（subject + job_id）
  3. 信息素驱动的自动派发

五、成功指标

指标目标值测量方法
派发成功率 ≥95% 成功派发数 / 总job数
验收准确率 ≥90% 人工抽查100个job，LLM判断与人工判断一致率
超时回收及时性 ≤5分钟 从超时到回收的时间间隔
心跳监控延迟 ≤2分钟 Worker离线到标记silent的时间
批量任务完成时间 100个job ≤ 8小时 从第一个派发到最后一个验收通过

六、风险与应对

风险影响应对措施
NATS单点故障 高 - 整个系统瘫痪 使用JetStream持久化 + 配置NATS集群（3节点）
LLM验收不稳定 中 - 误判导致返工 硬阈值优先（字数/文件数） + 人工抽查校准
Worker全部离线 中 - 任务堆积 发送告警通知用户 + job留在created等待
Job meta格式不规范 低 - 派发失败 Schema验证 + 错误日志记录

七、开放问题

  • Q1: acceptance标准如何标准化？能否提供模板？
    A1: Phase 1先支持自然语言，Phase 2提供常见模板（字数+文件数+关键词）
  • Q2: 多个Commander如何协同？
    A2: Phase 1只支持单Commander，Phase 4考虑主从模式或信息素驱动
  • Q3: Worker如何上报当前负载（CPU/内存）？
    A3: Phase 2在心跳中添加system_load字段
  • Q4: 如何处理依赖关系（JOB-048依赖JOB-047）？
    A4: Phase 3添加depends_on字段 + 拓扑排序派发
