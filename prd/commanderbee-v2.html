CommanderBee 完整设计 v2

排班经理 + 来料验收 + 蜂群协调 · 基于 Worker Bee 五层架构

一、核心定位

CommanderBee 是一个完整的 Worker Bee，共享五层工厂架构，但专注于监工和派发：

维度Worker BeeCommanderBee
架构五层工厂（完整）五层工厂（完整）
协议内核protocols.py + loop.pyprotocols.py + loop.py（复用）
Deck 机制按 Skill 动态装填按 Skill 动态装填
主要职能执行任务派发任务 + 验收成果
LLM 用途推理 + 工具调用汇总汇报 + 验收判断
专属 Skilllark-messaging, code-review...commander-dispatch, commander-acceptance...
NATS 角色接收任务 + 汇报状态派发任务 + 监控心跳

关键洞察：CommanderBee 不是一个独立的新框架。它是 Worker Bee 的一个特殊实例，通过不同的 Skill 集合和 Cronjob 配置，实现监工职能。

二、架构复用

共享的五层

五层：蜂群装备（NATS + Cron + Job Probe + SessionDB） ↓ 四层：Agent 外壳（~/.commander-bee/ 配置） ↓ 三层：协议内核（protocols.py + loop.py） ↓ 二层：工具边界（Deck + Registry + Skills） ↓ 一层：工具实现（swarm, file, terminal...）

差异点

Skills 目录skills/skills/（共享） + commander-skills/
agent.md执行者人格监工人格
soul.md友好助手公正裁判
Job 目录jobs/（接单）commander-jobs/（派单池）
Cronjob自用定时任务心跳监控 + 派发扫描 + 超时介入

三、LLM 使用边界

CommanderBee 是规则为主、LLM 为辅的混合架构：

场景用什么示例
Skill 匹配 纯规则 required_skills: [hayes-method-reviewer] → 查 bees.json 的 skills 列表
Bee 筛选 纯规则 status=alive + current_jobs < max_concurrent
超时回收 纯规则 job_accept 超时 + bee silent → 清空 owner
成果验收 LLM 读 acceptance: 三页互链完整 + 论证链清晰 + Worker 提交的 artifacts → 判断是否合格
多任务汇总 LLM 100 个 job 的状态 → 生成人类可读的周报
blocked 介入 LLM Worker 说"缺原文 PDF" → 判断：重派 / 等人工 / 修改 job
字数硬阈值 纯规则 字数 < 一半 → 自动重派（不问 LLM）

分界线：凡是能写成 if-else 的，用规则。凡是需要"理解语义"的（验收标准、汇总报告、blocked 原因分析），用 LLM。

四、专属 Skill 清单

1. commander-dispatch

---
name: commander-dispatch
description: 派发任务到 Worker Bee 蜂群
triggers:
  - 派发
  - 分配任务
  - dispatch job
tools:
  - fs_read_file      # 读 job meta.md
  - swarm_publish     # 发 swarm.task.new-job
  - fs_write_file     # 更新 job owner + phase
---

## 职责

读取 commander-jobs/ 下的 created 状态 job，匹配 Bee 能力，派发。

## 派发规则

1. 读 job frontmatter: `required_skills`
2. 读 bees.json: 筛选 `skills` 包含 required_skills
3. 筛选 `status=alive + current_jobs < max_concurrent`
4. 优先级：last_heartbeat 最近的（最近活跃优先）
5. 写入 `owner` + `phase=dispatched` → NATS 发 `swarm.task.new-job`

## 无匹配时

留在 created，等人加 Bee 或加 skill。不强行派发。

2. commander-acceptance

---
name: commander-acceptance
description: 验收 Worker Bee 提交的成果
triggers:
  - 验收
  - 审查结果
  - check deliverables
tools:
  - fs_read_file
  - fs_search_files
  - swarm_publish
---

## 职责

读 job 的 `acceptance` 标准，检查 Worker 提交的 artifacts。

## 验收步骤

1. 读 job meta.md: `acceptance` 字段（自然语言）
2. 读 job artifacts/ 目录: 列出所有文件
3. 读关键文件内容（lesson.html, notes.html...）
4. 判断：
   - ✅ Pass: 完全符合 acceptance
   - ⚠️ Needs work: 缺少部分内容，可补救
   - ❌ Incomplete: 严重不符，需重做

## 硬阈值（不问 LLM）

- 字数 < 一半 → 自动 ❌ + 重派
- 文件缺失（约定 3 个文件，只有 1 个）→ 自动 ❌

## 输出

写入 job `phase=reviewed` + 验收报告 → NATS 通知用户

3. commander-monitor

---
name: commander-monitor
description: 监控 Bee 心跳和 job 超时
triggers:
  - 监控
  - check status
  - monitor bees
tools:
  - fs_read_file
  - fs_write_file
  - swarm_publish
---

## 职责

扫描 bees.json + commander-jobs/，检测超时和阻塞。

## 检测规则

1. **Bee 心跳超时**（默认 2 分钟）
   - last_heartbeat 超时 → 标记 `status=silent`
   - 如果有 current_jobs → 回收 job（清空 owner）

2. **Job accept 超时**（默认 10 分钟）
   - phase=dispatched + 10 分钟没收到 job-started → 可能 Bee 停机
   - 清空 owner → phase=created

3. **Job execute 超时**（默认 4 小时）
   - phase=executing + 4 小时没收到 job-done → 可能卡住
   - 发 `swarm.command.rescind`

## 触发方式

Cronjob 每 60 秒自动运行。

4. commander-report

---
name: commander-report
description: 汇总蜂群状态，生成人类可读报告
triggers:
  - 汇报
  - 周报
  - status report
  - 看看进度
tools:
  - fs_read_file
  - fs_search_files
---

## 职责

读 commander-jobs/ 所有 job，生成汇总报告。

## 报告格式

### 蜂群状态
- 在线 Bee：N 台（bee-mac, bee-linux...）
- 繁忙 Bee：M 台
- 离线 Bee：K 台（超时 > 2 分钟）

### 任务进度
- 待派发：X 个（created）
- 执行中：Y 个（executing）
- 已完成：Z 个（done）
- 阻塞中：W 个（blocked）

### 最近完成（Top 5）
- JOB-047: 拆解论文 #047 · ✅ 2026-06-24 12:30
- ...

### 需要注意
- JOB-048: 超时 4 小时，已介入
- bee-linux: 心跳丢失 10 分钟

## 触发方式

- 用户问"看看进度" → 即时生成
- Cronjob 每天 9:00 自动发送周报

5. commander-intervention

---
name: commander-intervention
description: 处理 Worker Bee 的 blocked 消息
triggers:
  - blocked
  - 卡住了
  - intervention
tools:
  - fs_read_file
  - fs_write_file
  - swarm_publish
  - send_message
---

## 职责

读 Worker 发来的 `swarm.event.blocked`，决定介入策略。

## 介入决策（LLM）

读 blocked reason + job meta + 当前环境 → 判断：

1. **重新派发**
   - reason: "Bee 能力不足"
   - 清空 owner → phase=created → 等更强的 Bee

2. **等人工**
   - reason: "缺少原文 PDF"
   - 发消息通知用户 → phase=blocked

3. **修改 job**
   - reason: "acceptance 标准不清晰"
   - 建议用户明确标准

4. **自动补救**
   - reason: "依赖的 JOB-046 还没完成"
   - 等 JOB-046 done 后自动重派

## 输出

介入记录写入 job events + NATS 通知用户

六、NATS 消息协议

✨ 2026-06-24 更新：NATS 已升级为 JetStream 持久化系统。消息保留90天，listener重启不丢消息，支持durable pull consumer。配置需添加 server_name（集群必需）。

方向subjectpayload用途
CB → WB swarm.task.new-job {job_id, owner, meta_path} 派发新任务
CB → WB swarm.command.rescind {job_id, reason} 超时回收
CB → WB swarm.command.amend {job_id, field, new_value} 修改任务
WB → CB swarm.event.job-started {job_id, bee_id} Worker 开始干活
WB → CB swarm.event.job-done {job_id, bee_id, summary} Worker 完成任务
WB → CB swarm.event.blocked {job_id, bee_id, reason} Worker 遇到阻塞
WB → CB swarm.heartbeat.<bee-id> {status, current_jobs, skills} 心跳 + 能力上报

九、实施路径

Phase 1: 最小可用版本（MVP）

  1. 复制 Worker Bee 配置到 ~/.commander-bee/
  2. 写 3 个 Skill: commander-dispatch, commander-monitor, commander-acceptance
  3. 写 bees.json 维护逻辑（心跳注册）
  4. 配置 2 个 Cronjob: 心跳监控 + 派发扫描
  5. 手动测试：创建 job → Commander 派发 → Worker 执行 → Commander 验收

Phase 2: 自动化介入

  1. 实现 commander-intervention Skill
  2. 超时回收逻辑（job_accept / job_execute）
  3. blocked 消息处理

Phase 3: 人类友好界面

  1. 实现 commander-report Skill
  2. 每日周报 Cronjob
  3. Web dashboard（可选，后期）

Phase 4: WorldBee 接管

  1. WorldBee 负责 job 文件的物理投放
  2. CommanderBee 只发敲门铃（subject + job_id）
  3. 信息素驱动的自动派发（不再需要 Commander 显式调用）

十、关键设计原则

原则含义
架构复用 CommanderBee = Worker Bee + 不同的 Skill 集合
规则为主 能写 if-else 的不用 LLM
LLM 为辅 验收、汇总、介入决策用 LLM
唯一派单口 只有 Commander 能发 swarm.task.new-job
Cronjob 驱动 不需要 daemon，用 Cronjob 周期性唤醒
Markdown 状态 所有状态在文件里，人可读可编辑
NATS 解耦 Commander 和 Worker 只通过 NATS 通信
